## Findings

### emit-global-prompt.test.ts

- **Location:** `emit-global-prompt.test.ts` › `Cursor Plugin Emission` › `filename sanitization` › `should handle special characters in filenames` (line 151)
- **Smell:** `vacuous-assertion`
- **Rationale:** The test feeds `'My Rules (v2).md'` as both `name` and `sourcePath` and the title claims the SUT should *handle* the special characters. The only oracle is `expect(filename).toMatch(/^[\w-]+\.mdc$/)`, which is satisfied by any sanitizer that produces any string of word-chars/hyphens — including `'r.mdc'`, `'x-x.mdc'`, or a sanitizer that silently throws away every input and returns `'rule.mdc'` (which is the documented fallback, not the correct handling for this input). Matches the canonical signal "regex / prefix-only check where an exact value is knowable" from <https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/>.
- **Prescribed remediation:** Strengthen to a structural match on the *expected* sanitized name, e.g. `expect(filename).toBe('My-Rules-v2.mdc')` (or the project's documented sanitization output), so a mutant that returns the fallback or truncates the input would die. Per the canonical Prescribed Fix: prefer structural equality over regex/prefix; the kill-set must strictly increase.
- **Why this isn't a false positive:** This is not the "two-stage assertion required by the language" guard — there is no narrowing involved, the regex *is* the only oracle. The sanitized output is a knowable, deterministic string, so the weak regex is genuine under-oracling, not a documented absence-of-behavior contract.

### emit-skills.test.ts

- **Location:** `emit-skills.test.ts` › `Cursor Skills Emission` › `SimpleAgentSkill emission to .cursor/skills/` › `should sanitize skill names for directory creation` (line 71)
- **Smell:** `vacuous-assertion`
- **Rationale:** The skill `name` is `'My Skill (v2)'`. The test claims to verify sanitization for directory creation, but the only oracle on the produced directory is `expect(entries[0]).toMatch(/^[\w-]+$/)`. Any sanitizer producing any word-character string would pass — including one that returns a constant `'skill'`, one that drops every input down to a single letter, or one that fails to sanitize but happens to keep only word chars. Matches the canonical signal "structural-only check where an exact value is knowable" from <https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/>.
- **Prescribed remediation:** Replace the regex with a concrete `expect(entries[0]).toBe('My-Skill-v2')` (or the project's documented sanitization). If the exact mapping is intentionally implementation-defined, at minimum assert that the sanitized name retains a recognizable transformation of the input (e.g. `expect(entries[0]).toMatch(/skill/i)`) so a constant-fallback mutant dies.
- **Why this isn't a false positive:** Not "side-effect absence is the contract" — the contract is the *positive* derivation of a directory name from a noisy input. The current oracle does not encode that derivation; many wrong derivations still pass.
