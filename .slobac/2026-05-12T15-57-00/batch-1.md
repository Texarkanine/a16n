## Findings

### `discover-classification-priority.test.ts`

---

- **Location:** `discover-classification-priority.test.ts:18` — `should classify rules with globs as FileRule`
- **Smell:** `conditional-logic`
- **Rationale:** The test body is `for (const item of result.items) { expect(item.type).toBe(CustomizationType.FileRule); }` with no prior assertion on `result.items.length` and no warning/error check. Matches the canonical signal "Loops that assert inside but never on the loop's exit condition" — if `result.items` is empty (e.g. fixture removed, classifier silently drops everything, glob matcher mis-fires) the loop body never runs and the test is green by omission. The "should classify rules with globs as FileRule" claim is silently satisfied by zero rules. See https://texarkanine.github.io/slobac/taxonomy/conditional-logic/.
- **Prescribed remediation:** Per the canonical "Loop without exit assertion" row of the Prescribed Fix table: assert on the collected results after (or before) the loop. Here, add `expect(result.items.length).toBeGreaterThan(0);` (or, better, `expect(result.items).toHaveLength(N);` matching the fixture's known rule count) before the loop, so the loop's vacuous-iteration path becomes a failing test.
- **Why this isn't a false positive:** Could be confused with the symmetric-parameterized-table guard, but this is not a parameterized table — there is no per-row expected/actual pairing, just a uniform assertion over an iteration whose count is unverified. The test is not using a runner-native skip, has no try/catch, and the iteration count is a load-bearing precondition that is never checked.

---

- **Location:** `discover-classification-priority.test.ts:27` — `should classify rules with description (no globs) as SimpleAgentSkill`
- **Smell:** `conditional-logic`
- **Rationale:** Identical shape to the L18 finding: `for (const item of result.items) { expect(item.type).toBe(CustomizationType.SimpleAgentSkill); }` with no length assertion before or after. If the `cursor-agentskill` fixture is empty or the discover step returns no items, the test passes silently. The signal "Loops that assert inside but never on the loop's exit condition" matches verbatim. See https://texarkanine.github.io/slobac/taxonomy/conditional-logic/.
- **Prescribed remediation:** Add a length assertion gating the loop (e.g. `expect(result.items).toHaveLength(2)` matching the fixture, mirroring how the sibling file `discover-simple-agent-skill-rules.test.ts:14` already does it).
- **Why this isn't a false positive:** Not a try/catch-with-fail and not a runner-native skip. The iteration is over a list whose minimum size is part of the contract under test; the absence of that size check is the smell.

---

- **Location:** `discover-classification-priority.test.ts:47` — `should classify rules with valid globs over description (globs takes precedence)`
- **Smell:** `conditional-logic`
- **Rationale:** Same loop-without-exit-assertion shape as L18 and L27. Body: `for (const item of result.items) { expect(item.type).toBe(CustomizationType.FileRule); }` with no length precondition. The test's claim is "globs takes precedence over description" — but if `result.items` is empty, the precedence claim is unverified and the test is green. See https://texarkanine.github.io/slobac/taxonomy/conditional-logic/.
- **Prescribed remediation:** Add `expect(result.items).toHaveLength(2)` (the `cursor-filerule` fixture has two rules, per the sibling assertion at L14 of the same file) before the loop. Even better, assert directly on the rule that has *both* globs and description, since that's the precedence case the title actually claims.
- **Why this isn't a false positive:** Not a parameterized matrix; not a runner skip. The "precedence" semantics are the contract, and an empty-items pass would not detect a regression that broke classification entirely.

### `discover-cursor-plugin.test.ts`

---

- **Location:** `discover-cursor-plugin.test.ts:119` — `should set relativeDir on all rule types (GlobalPrompt, FileRule, SimpleAgentSkill)`
- **Smell:** `vacuous-assertion`
- **Rationale:** The test's claim is that `relativeDir` is *set* on all three rule types. The actual oracle for `relativeDir` is three `expect(rule?.relativeDir).toBeDefined()` calls (lines 135–137). Per the canonical signal "`assert x is not None` ... as the *only* assertion" and the example contrasting `toBeDefined()` with a structural matcher, this oracle accepts wrong answers that the title rules out: `relativeDir = ""`, `relativeDir = "/"`, `relativeDir = "wrong/path"`, or any garbage string would all satisfy `toBeDefined()`. The neighboring test at L102 (`should set relativeDir for deeply nested rules…`) does assert exact values (`'shared/niko'`, `'shared/niko/Core'`, `'shared/niko/Level1'`) — proving the stronger oracle is available and easy. See https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/.
- **Prescribed remediation:** Replace the three `toBeDefined()` checks with the same expected-value assertions used at L114–L116 of the same file (`expect(mainRule?.relativeDir).toBe('shared/niko')`, etc.), or fold this test into L102 with an additional per-type assertion that the value is the expected nested path. Per the Prescribed Fix's preferred order: structural equality > matcher containment > regex > prefix/length.
- **Why this isn't a false positive:** Not "side-effect absence is the contract" — the contract here is a positive value (a path string). Not "two-stage required language narrowing" — there is no second-stage destructure that would constitute the real oracle; `toBeDefined()` *is* the only oracle on the `relativeDir` field. The `expect(rule?.type).toBe(...)` lines verify the *type* claim, not the relativeDir claim.
