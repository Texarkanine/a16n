### 1. `discover-classification-priority.test.ts:18` — `conditional-logic`

- **Location:** `discover-classification-priority.test.ts` → `should classify rules with globs as FileRule`
- **Smell:** `conditional-logic`
- **Rationale:** The test body is `for (const item of result.items) { expect(item.type).toBe(CustomizationType.FileRule); }` with no prior assertion on `result.items.length`. Matches the canonical signal "Loops that assert inside but never on the loop's exit condition" — if `result.items` is empty (fixture removed, classifier silently drops everything, glob matcher mis-fires) the loop body never runs and the test is green by omission. The "should classify rules with globs as FileRule" claim is silently satisfied by zero rules. See <https://texarkanine.github.io/slobac/taxonomy/conditional-logic/>.
- **Prescribed remediation:** Per the canonical "Loop without exit assertion" row of the Prescribed Fix table: assert on the collected results before the loop. Add `expect(result.items).toHaveLength(2)` (matching the fixture's two rules — see how the sibling file `discover-simple-agent-skill-rules.test.ts:14` already does it), so the loop's vacuous-iteration path becomes a failing test.
- **Why this isn't a false positive:** Could be confused with the symmetric-parameterized-table guard, but this is not a parameterized table — there is no per-row expected/actual pairing, just a uniform assertion over an iteration whose count is unverified. No try/catch, no runner-native skip; the iteration count is a load-bearing precondition that is never checked.

### 4. `discover-cursor-plugin.test.ts:119` — `vacuous-assertion`

- **Location:** `discover-cursor-plugin.test.ts` → `should set relativeDir on all rule types (GlobalPrompt, FileRule, SimpleAgentSkill)`
- **Smell:** `vacuous-assertion`
- **Rationale:** The test's claim is that `relativeDir` is *set* on all three rule types. The actual oracle for `relativeDir` is three `expect(rule?.relativeDir).toBeDefined()` calls (lines 135–137). Per the canonical signal "`assert x is not None` … as the *only* assertion" and the example contrasting `toBeDefined()` with a structural matcher: `relativeDir = ""`, `relativeDir = "/"`, `relativeDir = "wrong/path"`, or any garbage string would all satisfy `toBeDefined()`. The neighboring test at L102 (`should set relativeDir for deeply nested rules…`) does assert exact values (`'shared/niko'`, `'shared/niko/Core'`, `'shared/niko/Level1'`), proving the stronger oracle is available and easy. See <https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/>.
- **Prescribed remediation:** Replace the three `toBeDefined()` checks with the same expected-value assertions used at L114–L116 (`expect(mainRule?.relativeDir).toBe('shared/niko')`, etc.), or fold this test into L102 with an additional per-type assertion that the value is the expected nested path. Per the Prescribed Fix's preferred order: structural equality > matcher containment > regex > prefix/length.
- **Why this isn't a false positive:** Not "side-effect absence is the contract" — the contract here is a positive value (a path string). Not "two-stage required language narrowing" — there is no second-stage destructure that would constitute the real oracle; `toBeDefined()` *is* the only oracle on the `relativeDir` field. The `expect(rule?.type).toBe(...)` lines verify the *type* claim, not the relativeDir claim.

### 5. `emit-global-prompt.test.ts:151` — `vacuous-assertion`

- **Location:** `emit-global-prompt.test.ts` → `Cursor Plugin Emission` › `filename sanitization` › `should handle special characters in filenames`
- **Smell:** `vacuous-assertion`
- **Rationale:** The test feeds `'My Rules (v2).md'` as both `name` and `sourcePath` and the title claims the SUT should *handle* the special characters. The only oracle is `expect(filename).toMatch(/^[\w-]+\.mdc$/)`, which is satisfied by any sanitizer that produces any string of word-chars/hyphens — including `'r.mdc'`, `'x-x.mdc'`, or one that silently throws away every input and returns `'rule.mdc'` (which is the *documented fallback*, not the correct handling for this input). Matches the canonical signal "regex / prefix-only check where an exact value is knowable" from <https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/>.
- **Prescribed remediation:** Strengthen to a structural match on the *expected* sanitized name, e.g. `expect(filename).toBe('My-Rules-v2.mdc')` (or whatever the project's documented sanitization output is for this input), so a mutant that returns the fallback or truncates the input would die. Per the Prescribed Fix: prefer structural equality over regex/prefix; the kill-set must strictly increase.
- **Why this isn't a false positive:** This is not the "two-stage assertion required by the language" guard — there is no narrowing involved, the regex *is* the only oracle. The sanitized output is a knowable, deterministic string, so the weak regex is genuine under-oracling, not a documented absence-of-behavior contract.

### 6. `emit-skills.test.ts:71` — `vacuous-assertion`

- **Location:** `emit-skills.test.ts` → `Cursor Skills Emission` › `SimpleAgentSkill emission to .cursor/skills/` › `should sanitize skill names for directory creation`
- **Smell:** `vacuous-assertion`
- **Rationale:** The skill `name` is `'My Skill (v2)'`. The test claims to verify sanitization for directory creation, but the only oracle on the produced directory is `expect(entries[0]).toMatch(/^[\w-]+$/)`. Any sanitizer producing any word-character string would pass — including one that returns a constant `'skill'`, one that drops every input down to a single letter, or one that fails to sanitize but happens to keep only word chars. Matches the canonical signal "structural-only check where an exact value is knowable" from <https://texarkanine.github.io/slobac/taxonomy/vacuous-assertion/>.
- **Prescribed remediation:** Replace the regex with a concrete `expect(entries[0]).toBe('My-Skill-v2')` (or the project's documented sanitization). If the exact mapping is intentionally implementation-defined, at minimum assert that the sanitized name retains a recognizable transformation of the input (e.g. `expect(entries[0]).toMatch(/skill/i)`) so a constant-fallback mutant dies.
- **Why this isn't a false positive:** Not "side-effect absence is the contract" — the contract is the *positive* derivation of a directory name from a noisy input. The current oracle does not encode that derivation; many wrong derivations still pass.

See <https://texarkanine.github.io/slobac/taxonomy/semantic-redundancy/>.
- **Prescribed remediation:** **Keep** `discover-file-rule.test.ts:10` (canonical — it lives in the file dedicated to FileRule discovery and has the strongest oracle: `expect(result.items).toHaveLength(2)` before the iteration, killing the empty-iteration mutant). **Delete** `discover-classification-priority.test.ts:18` as a strict subset (its weaker oracle is what finding 1 above flagged as `conditional-logic`). Not "same surface, different business concept" — the L47 title *promises* a different concept (precedence) but the body and fixture do not encode it; the actually-verified observable is identical to the other two. Not "contract-duplication of production constants" — no inline literal whose production drift is being guarded.
